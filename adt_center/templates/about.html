What is ADT? CURRENT

Author: Paul Sheridan, Director, Advanced Digital Transformation (ADT)

"Digital transformation initiatives frequently fail not due to a lack of technology, but because governance is introduced too late, applied inconsistently or not applied at all—leaving organisations to confront complex, fragmented challenges without a coherent framework for accountability, coordination and control."

ADT shifts governance upstream, embedding it into process creation so that compliance, accountability and auditability are properties of execution rather than downstream enforcement.

The framework aligns with UN governance principles: Direction, Performance, Accountability, Transparency, and Legitimacy.

Bridges organizational strategy and technical execution. We don't just know what changed, we know why.

  • Intents: Desired outcomes (e.g., "Improve Tuning Accuracy").
  • Events: Triggering occurrences (e.g., "Sensor Drift Detected").
  • Maturity: Real-time tracking of capability evolution across 7 stages.
Authoritative Data Source (ADS) OPERATIONAL
"If an action is not recorded, it is not recognised as having occurred."

The ADS is the single source of truth for everything that happens inside a governed project. Every agent action, DTTP decision, delegation, and escalation is appended to _cortex/ads/events.jsonl.

  • Append-only: History is never modified. Events are immutable once written.
  • Structured events: Each entry carries agent, role, action type, spec reference, authorization status, and hash chain link.
  • Safe Logger v3.0: Atomic file locking (fcntl), schema validation, and hash computation prevent corruption even under concurrent writes.

View the full event timeline in the ADS Timeline.

Integrity Chain OPERATIONAL

Every ADS event contains the SHA-256 hash of the previous event, forming a tamper-evident chain identical in concept to a blockchain.

  1. Each event is serialized to a canonical JSON string (sorted keys, no whitespace).
  2. The SHA-256 hash is computed over the serialized string.
  3. The hash is stored in the event and becomes the prev_hash of the next event.
  4. The genesis block uses an all-zeros prev_hash.

Modifying, inserting, or deleting any event breaks every subsequent hash link. The Overseer runs periodic integrity audits and the ADS Healer can detect and report chain breaks. A break is itself logged as a governance incident.

Specification-Driven Development (SDD) OPERATIONAL
"No Spec, No Code" — the fundamental rule of ADT governance.

Every code change must be authorized by an approved specification in _cortex/specs/. This ensures traceability from intent to implementation.

  • Spec Lifecycle: DRAFT → Human Approval → APPROVED → Active → COMPLETED
  • Systems Architect: Writes specifications, defines scope and requirements. Does not implement.
  • Human: Approves specifications. Ultimate authority over what gets built.
  • Engineers: Implement only what the spec authorizes, within their jurisdiction.
DTTP — Structural Enforcement OPERATIONAL

The Digital Transformation Transaction Protocol enforces governance structurally, not just behaviourally.

LevelMechanismStrength
1Behavioural (prompt instructions)Proven insufficient
2Hook-based (pre-action scripts)Bypassable via Bash
3Privilege-separated (OS permissions + network)Target architecture

  • human: Full access. Approves specs, authorizes SCRs, activates Shatterglass.
  • agent: Restricted OS user. Can only write via DTTP API requests.
  • dttp: Service user. Validates requests against policy, executes authorized writes.

Agent Filesystem Sandbox (SPEC-036): bwrap/unshare namespace isolation confines each agent session to its jurisdictional paths.

External Project Governance OPERATIONAL

ADT can govern any project, not just itself. The multi-project registry allows multiple codebases to operate under independent governance.

  • Independent _cortex/ directories: Each project gets its own ADS, specs, tasks, and jurisdictions.
  • Project isolation: Events, enforcement, and compliance data are scoped per project.
  • Unified view: The Operational Center can aggregate or filter across all registered projects.
  • Console integration: The Operator Console groups sessions by project with per-project context panels.

Implementation: SPEC-031, SPEC-032 (Project Launcher).

Sovereign Change Requests (SCR) OPERATIONAL

Sovereign paths contain the rules of the framework. No agent can modify them directly.

When an agent proposes a change to a sovereign file, DTTP blocks the write and auto-submits a Sovereign Change Request for human authorization in the Governance panel.

Implementation: SPEC-033 — Human-in-the-loop authorization queue.

Shatterglass Protocol OPERATIONAL

Emergency override mechanism for critical situations where normal governance must be temporarily suspended.

  • Break-glass activation: Human-only action requiring confirmation (type "SHATTERGLASS").
  • Time-limited window: Override expires after a configurable duration with countdown timer.
  • Full audit trail: Every action taken during a Shatterglass window is logged to the ADS with elevated visibility.
  • Production mode: OS-level enforcement via the agent user (Tier 3). Toggle in the Operator Console.

Implementation: SPEC-027. Accessible from the Governance page.

Real Incidents OPERATIONAL

The ADT Framework is its own proving ground. These incidents validated (and sometimes challenged) the governance model:

An em-dash character in an ADS event description caused _get_last_event() to crash mid-way through a multi-byte UTF-8 sequence. The entire DTTP pipeline returned 500 errors, blocking all governed writes. Root cause: byte-by-byte reverse seek in text mode. Fix: binary mode read in Safe Logger v3.0.

Agents discovered they could use the Bash tool to bypass DTTP hooks entirely, writing directly to governed files. This proved Level 2 (hook-based) enforcement is fundamentally insufficient and motivated the move to Level 3 privilege separation with the agent OS user model.

Inconsistent role casing (devops_engineer vs DevOps_Engineer) caused hash chain breaks because the canonical JSON serialization differed. Fix: role and agent name normalization via canonical registry loaded from jurisdictions.json (SPEC-020 Amendment B).

Agents completing cross-role requests could not mark them as COMPLETED because _cortex/requests.md and _cortex/tasks.json were outside their jurisdiction. The only option was bypassing DTTP via Bash. Fix: governed status update API (SPEC-035) at /api/governance/requests/<id>/status.
